## 1. Теоретичні відомості

### Структура обробки виключень

```python
try:
    # код, що може викликати помилку
except ТипПомилки as e:
    # обробка конкретної помилки
except (TypeError, ValueError):
    # кілька типів в одному блоці
except Exception as e:
    # будь-яке виключення
else:
    # виконується ЛИШЕ якщо виключень не було
finally:
    # виконується ЗАВЖДИ (очищення ресурсів)
```

---

### Стандартні виключення Python

| Виключення | Причина |
|---|---|
| `ZeroDivisionError` | Ділення на нуль |
| `ValueError` | Некоректне значення (напр. `int("abc")`) |
| `TypeError` | Неправильний тип даних |
| `IndexError` | Вихід за межі списку |
| `KeyError` | Відсутній ключ словника |
| `FileNotFoundError` | Файл не знайдено |
| `PermissionError` | Немає прав доступу |
| `AttributeError` | Об'єкт не має атрибута |
| `RecursionError` | Перевищено ліміт рекурсії |
| `OverflowError` | Переповнення числового типу |

---

### Власні виключення

```python
class AppError(Exception):
    pass

class InsufficientFundsError(AppError):
    def __init__(self, balance, amount):
        super().__init__(f"Баланс {balance}, потрібно {amount}")
        self.balance = balance
        self.amount = amount

raise InsufficientFundsError(200, 500)
```

Власні виключення успадковують `Exception` і дозволяють передавати додаткові дані.

---

### Ланцюжок виключень

```python
try:
    int(data["key"])
except KeyError as e:
    raise DataError("Відсутній ключ") from e  # зберігає оригінальну причину
```

## 3. Висновки

У процесі виконання практичної роботи було освоєно:

- **Блок `try/except`** перехоплює помилки і не дає програмі впасти. Завжди варто перехоплювати конкретні типи, а не голий `except`, щоб не приховати непередбачені помилки.
- **Блок `else`** виконується тільки за відсутності виключень — зручно відокремити «успішний» код від коду обробки помилок.
- **Блок `finally`** гарантує виконання незалежно від результату (закриття файлів, звільнення з'єднань). Завжди використовувати для ресурсів.
- **Кілька `except`** дозволяють по-різному реагувати на різні помилки. Порядок важливий — вужчі типи мають іти перед ширшими.
- **Власні виключення** роблять код читабельним і дозволяють передавати структуровану інформацію про помилку (поле, значення, контекст).
- **Ланцюжок `raise ... from`** зберігає оригінальну причину помилки, що спрощує діагностику.
- **Retry-патерн** (повторні спроби) реалізується через цикл із `try/except` і є стандартним підходом для мережевих операцій.

Правильна обробка виключень є ознакою production-готового коду: програма залишається стабільною навіть при некоректних вхідних даних або зовнішніх збоях.
